![](http://oa.htek.com:8090/download/attachments/257003812/image2026-4-10_18-13-40.png?version=1&modificationDate=1775816020215&api=v2)![](http://oa.htek.com:8090/download/attachments/257003812/image2026-4-10_18-20-16.png?version=1&modificationDate=1775816416552&api=v2)

## 网页上2个调参说明：

## 1) CFG 是什么（`cfg_scale`）

它控制的是 Classifier-Free Guidance 强度，在扩散采样里直接参与公式

* `cfg_scale` 越大：模型越“听条件”（文本/上下文）
* `cfg_scale` 越小：模型更自由，可能更自然但也可能偏离文本意图

### 体感影响（你这个实时TTS场景）

* 增大 CFG：文本跟随更强、语气更“硬”、有时不够自然
* 减小 CFG：更柔和，但可能发音/停顿更漂、偶发不稳定

---

## 2) 推理步数是什么（`inference_steps`）

它对应扩散采样步数（DDPM inference steps），在采样前被设置，采样时会按这个步数循环：

* 步数越多：通常细节更好、噪声更低，但更慢
* 步数越少：更快，但细节/稳定性可能下降

---

## 3) 两者一起怎么理解（最重要）

* `CFG` 决定“往哪走”（条件引导强弱）
* `Steps` 决定“走几步”（采样精细度和耗时）

可以理解成：  
Text + Voice Promptcfg\_scale 控制引导强度每一步扩散更新inference\_steps 决定步数语音token音频输出

---

## 4) 结合你当前机器（GTX 1060）调参建议

你现在是老卡兼容路径（更偏稳定优先），建议：

* 先保守：`CFG=1.5`, `Steps=5`（你当前默认）
* 文本跟随不够时：`CFG 1.7 ~ 2.0`
* 音质想更稳一点：`Steps 6 ~ 8`（速度会明显变慢）
* 如果出现卡顿/首包慢：先把 `Steps` 降回 `5`，再调 `CFG`

一个实用顺序：

1. 固定 `Steps=5`，先找最顺耳的 `CFG`
2. 再小幅加 `Steps` 到 `6/7` 看是否值得（质量提升 vs 延迟）

# 三、文档 VibeVoice-Realtime（中文版）：支持流式输入的实时长语音 TTS

## [🎙️ VibeVoice-Realtime（中文版）：支持流式输入的实时长语音 TTS](#-vibevoice-realtime中文版支持流式输入的实时长语音-tts)

[![Hugging Face](https://img.shields.io/badge/HuggingFace-Collection-orange?logo=huggingface)](https://huggingface.co/microsoft/VibeVoice-Realtime-0.5B) [![Colab](https://img.shields.io/badge/Run-Colab-orange?logo=googlecolab)](https://colab.research.google.com/github/microsoft/VibeVoice/blob/main/demo/vibevoice_realtime_colab.ipynb)

`VibeVoice-Realtime` 是轻量级实时文本转语音模型，支持**流式文本输入**与**长语音稳健生成**。首段可听语音延迟约 **200ms**（取决于硬件）。

**模型：** [VibeVoice-Realtime-0.5B](https://huggingface.co/microsoft/VibeVoice-Realtime-0.5B) **Colab：** [Link](https://colab.research.google.com/github/microsoft/VibeVoice/blob/main/demo/vibevoice_realtime_colab.ipynb)

> 说明：模型以英文能力为主。官方也提供德/法/意/日/韩/荷/波/葡/西等语种供探索，但未做充分验证。

## [特性](#特性)

* 参数规模：0.5B（部署友好）
* 实时 TTS（首段低延迟）
* 流式文本输入
* 长语音生成（8K context，约 10 分钟）

## [架构说明](#架构说明)

该实时版本采用交错窗口设计：一边增量编码新文本，一边基于历史上下文继续声学 latent 的扩散生成。与多说话人长文本版本不同，该模型移除了语义 tokenizer，仅保留高效声学 tokenizer（7.5 Hz）。

## [使用限制](#使用限制)

* 当前仅支持**单说话人**实时生成。
* 多说话人对话请使用其他 VibeVoice 变体。
* 非英文输入可能出现不可预期结果。

## [安装](#安装)

```
# 推荐容器环境（示例）
sudo docker run --privileged --net=host --ipc=host --ulimit memlock=-1:-1 --ulimit stack=-1:-1 --gpus all --rm -it nvcr.io/nvidia/pytorch:24.07-py3

git clone https://github.com/microsoft/VibeVoice.git
cd VibeVoice/
pip install -e .[streamingtts]
```

## [离线环境变量说明（Windows PowerShell）](#离线环境变量说明windows-powershell)

你常用的两条命令：

```
$env:HF_HUB_OFFLINE="1"
$env:TRANSFORMERS_OFFLINE="1"
```

### [1) 这两个变量是控制什么的](#1-这两个变量是控制什么的)

* `HF_HUB_OFFLINE=1`：让 Hugging Face Hub 进入离线模式，不再访问 `huggingface.co` 拉取/校验文件。
* `TRANSFORMERS_OFFLINE=1`：让 `transformers` 优先只用本地缓存，不再发起在线下载请求。

在本项目里，设置后可避免启动时卡在 `tokenizer_config.json` 等网络超时重试。

### [2) 如何使用](#2-如何使用)

#### [方式 A：当前终端会话临时生效（推荐）](#方式-a当前终端会话临时生效推荐)

```
$env:HF_HUB_OFFLINE="1"
$env:TRANSFORMERS_OFFLINE="1"
python .\VibeVoice\demo\vibevoice_realtime_demo.py --model_path "C:\Htek\ARSTTS\models\VibeVoice-Realtime-0.5B" --device cuda --port 3000
```

#### [方式 B：一行写法](#方式-b一行写法)

```
$env:HF_HUB_OFFLINE="1"; $env:TRANSFORMERS_OFFLINE="1"; python .\VibeVoice\demo\vibevoice_realtime_demo.py --model_path "C:\Htek\ARSTTS\models\VibeVoice-Realtime-0.5B" --device cuda --port 3000
```

### [3) 使用效果](#3-使用效果)

* 优点：
* 启动更稳定，不受外网波动影响；
* 避免 HF 超时重试导致的“卡一会”。
* 注意：
* 必须保证所需模型与 tokenizer 文件已在本地缓存或本地模型目录中；
* 若本地缺文件，离线模式会直接报“找不到本地文件”，不会自动下载。

### [4) 使用后如何恢复联网](#4-使用后如何恢复联网)

仅对**当前 PowerShell 窗口**恢复：

```
Remove-Item Env:HF_HUB_OFFLINE -ErrorAction SilentlyContinue
Remove-Item Env:TRANSFORMERS_OFFLINE -ErrorAction SilentlyContinue
```

或者直接设回 `0`：

```
$env:HF_HUB_OFFLINE="0"
$env:TRANSFORMERS_OFFLINE="0"
```

也可以直接关闭当前终端并新开一个终端（环境变量会话级失效）。

## [风险与限制](#风险与限制)

* 可能出现偏差、错误或不可预期输出。
* 存在深度伪造与误导传播风险，需合规使用。
* 主要面向英文；其他语言结果不稳定。
* 不支持代码/公式/特殊符号朗读。
* 输入过短（<=3 词）时稳定性可能下降。
