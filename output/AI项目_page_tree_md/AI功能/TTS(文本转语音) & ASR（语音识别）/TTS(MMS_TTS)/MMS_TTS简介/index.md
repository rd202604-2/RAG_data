# [MMS-TTS 详细介绍（面向 RK3576 / Android 14 落地）](#mms-tts-详细介绍面向-rk3576--android-14-落地)

本文档解释 `MMS-TTS` 是什么、模型怎么工作、和常见 TTS 的差异、在 RK3576 上落地时要注意什么。 配套实操命令见本目录中的《RK3576 + Android 14 + MMS-TTS 最短命令清单》。

---

## [1. MMS-TTS 是什么](#1-mms-tts-是什么)

`MMS` 全称 **Massively Multilingual Speech**（Meta 发布的多语言语音项目），其中 `MMS-TTS` 是其文字转语音分支。 你在 Hugging Face 上常见到类似模型名：

* `facebook/mms-tts-eng`
* `facebook/mms-tts-deu`
* `facebook/mms-tts-fra`

即：**每个语言一个模型（或一组配置）**，不是“单模型全语种高质量统一音色”。

---

## [2. MMS-TTS 的定位](#2-mms-tts-的定位)

### [2.1 适合场景](#21-适合场景)

* 需要快速支持多语种
* 对“先能说出来”要求高于“主播级音色”
* 端侧/边缘设备想优先选择小模型与可部署性

### [2.2 不适合场景](#22-不适合场景)

* 强情感、强韵律、超自然拟人效果
* 商业级品牌音色（通常要更大模型或定制训练）
* 单模型统一几十语种并保持稳定高质量

---

## [3. 典型推理链路（工程视角）](#3-典型推理链路工程视角)

在 RKNN 生态（`rknn_model_zoo/examples/mms_tts`）里，MMS-TTS 常被拆成两个阶段：

1. **文本/符号序列 -> 声学中间表示**（常见为 encoder 侧）
2. **中间表示 -> 波形或声码器输入 -> 音频输出**（常见为 decoder/vocoder 侧）

工程上你会看到两份或多份模型文件，例如：

* `encoder.onnx` / `encoder.rknn`
* `decoder.onnx` / `decoder.rknn`

这也是为什么文档与脚本里常出现“先导出 ONNX，再分别转 RKNN”。

---

## [4. 与你正在用的 VibeVoice-Realtime-0.5B 的核心区别](#4-与你正在用的-vibevoice-realtime-05b-的核心区别)

| 维度 | MMS-TTS | VibeVoice-Realtime-0.5B |
| --- | --- | --- |
| 模型目标 | 多语言覆盖、可部署 | 实时长文本流式、更强调连续生成体验 |
| 常见部署路径 | ONNX / RKNN / 端侧 CPU-NPU | Python + PyTorch（常见桌面/服务器） |
| 模型组织 | 常见按语言拆分模型 | 单模型能力集成 |
| 板端可行性 | 较常见，有 RKNN 样例可走 | 官方仓库无 RK3576 Android 直跑方案 |

你当前需求是 `RK3576 + Android 14`，优先 MMS-TTS 是合理路线。

---

## [5. 在 RK3576 上为何常选 MMS-TTS](#5-在-rk3576-上为何常选-mms-tts)

1. **已有 RKNN 示例基础**：`rknn_model_zoo/examples/mms_tts`
2. **模型规模与链路相对可控**：更容易做 ONNX -> RKNN
3. **多语言扩展清晰**：按语言包选择模型，替换成本低
4. **适合端侧裁剪优化**：便于逐步做量化与性能调优

---

## [6. 端到端文件流（从下载到上板）](#6-端到端文件流从下载到上板)

你在工程里一般会经历这条文件链路：

1. 从 Hugging Face 下载：`facebook/mms-tts-<lang>`
2. 导出 ONNX：`encoder.onnx` + `decoder.onnx`（名称按脚本实际）
3. 转 RKNN：`encoder.rknn` + `decoder.rknn`
4. `adb push` 到板端：`/data/local/tmp/mms_tts/`
5. 运行 demo，输出 `out.wav`

这条流程与本目录的“最短命令清单”文档是一一对应的。

---

## [7. 性能与质量的关键影响项](#7-性能与质量的关键影响项)

### [7.1 性能（延迟）](#71-性能延迟)

* 是否真正走到 NPU（不是 CPU 回退）
* 输入文本长度（句子越长，端到端耗时越高）
* `rknn_toolkit2` 与 `librknnrt.so` 的版本匹配情况

### [7.2 质量（可懂度/自然度）](#72-质量可懂度自然度)

* 语言模型是否选对（`eng/deu/...`）
* 文本预处理（标点、数字、缩写展开）
* 采样率与后处理（去爆音、响度归一）

---

## [8. 常见坑（RK3576 Android 14）](#8-常见坑rk3576-android-14)

### [8.1 版本不匹配](#81-版本不匹配)

症状：`RKNN init fail`、`invalid model version` 本质：PC 转换工具链与板端 runtime 版本不一致。 建议：统一用同一套 BSP/SDK 的 toolkit + runtime。

### [8.2 so 加载失败](#82-so-加载失败)

症状：`cannot locate symbol`、程序启动即崩。 本质：ABI 不一致或 `LD_LIBRARY_PATH` 缺失。 建议：确认 `arm64-v8a`，并在板端导出 `LD_LIBRARY_PATH`。

### [8.3 能跑但慢](#83-能跑但慢)

症状：首包慢、句子长时卡顿。 本质：未上 NPU、线程配置不合理、文本过长。 建议：先短句压测，再做分句推理与流水线缓存。

---

## [9. 与 Android App 集成建议（最小可行）](#9-与-android-app-集成建议最小可行)

建议分两阶段：

1. **阶段 A：先在 `adb shell` 跑通命令行 demo**

* 目标是验证模型、runtime、NPU 路径正确

1. **阶段 B：再做 App 封装**

* Java/Kotlin 负责 UI 与文本输入
* JNI/C++ 负责 RKNN 推理与 WAV 输出
* AudioTrack/MediaPlayer 负责播放

不要一开始就直接写完整 APK；先确保底层推理链路稳定。

---

## [10. 选型结论（给你的场景）](#10-选型结论给你的场景)

对于你当前的 `RK3576 + Android 14`：

* 想快速落地：优先 `MMS-TTS + RKNN Model Zoo` 路线
* 想中文商业效果：并行评估厂商离线 TTS SDK
* 想长期可控：保留 ONNX 中间层，便于后续替换模型

---
